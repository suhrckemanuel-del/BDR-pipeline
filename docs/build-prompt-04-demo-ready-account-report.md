```text
You are Codex working in:

Goal:
Create a demo-ready account report that packages the whole pipeline output into one founder-readable artifact. The viewer should understand, in one page:
- who the target account is
- why it is or is not worth reviewing
- what evidence the system found
- which contact/persona it recommends
- what outreach angle and sequence it drafted
- what risks the critic flagged
- what the human should do next

Product positioning:
This is a founder-safe AI BDR workflow. The report should make the project feel like a practical product, not a script frontend. It should prove judgment, evidence handling, multi-agent orchestration, and human approval. Do not claim reply rates, meetings, revenue, or campaign lift.

Feature boundary:
Implement the demo-ready account report only. Do not build the full review queue, auto-send, reply analytics, LLM council, CRM rewrite, eval harness, or new outbound channels in this pass.

Important principle:
The account report should be assembled deterministically from existing state. Do not add another LLM call just to write a pretty report. This report is a presentation layer over already-produced evidence, scores, strategy, sequence, and risk gate.

Repo map:
- Streamlit layout: app/ui/layout.py
- UI components: app/ui/components.py
- CSS: app/ui/theme.py
- Shared state/schemas: app/agents/state.py
- Existing account score: EnrichmentResult.account_score
- Existing evidence: EnrichmentResult.evidence_cards
- Existing strategy: state["strategy"]
- Existing sequence/drafts: state["card"]
- Existing quality gate: state["critic_result"].quality_gate

Implementation requirements:

1. Add a pure report builder.

Create a small module:
app/services/report_builder.py

It should expose deterministic helpers such as:
- build_account_report_markdown(state: dict, tenant: TenantConfig) -> str
- build_report_filename(company: str) -> str
- optional helper functions for safe labels, top evidence selection, recommended contact, and next action.

Keep this module pure:
- no Streamlit imports
- no network calls
- no LLM calls
- no API keys
- no filesystem writes

2. Report content.

The report should include these sections in Markdown:

Title:
# BDR Account Report: <Company>

Metadata:
- Industry
- Tenant / product
- Persona
- Generated by BDR Pipeline
- Manual review required before sending

Decision summary:
- Account score: X/100
- Priority: High Priority / Review / Needs More Research / Do Not Send Yet
- Gate verdict: Approved / Needs Edit / Needs More Research / Do Not Send Yet
- Safe to send: Yes/No
- Recommended next action

Why this account:
- ICP fit rationale
- Pain evidence rationale
- Trigger strength rationale
- Strategy recommended angle
- Strategy pain signal

Evidence highlights:
- 3-6 top evidence cards
- Show evidence_id, confidence, support type, claim, source title, source URL if available
- Separate observed/source-backed evidence from derived/internal scoring.

Recommended contact:
- name
- title
- email if present
- confidence
- LinkedIn URL if present
- warning if contact confidence is low or no contacts were found

Outreach recommendation:
- recommended angle
- entry persona
- 5-touch sequence summary
- include subject/body for email touches and body for LinkedIn touches
- keep formatting readable

Quality + risk:
- critic overall quality
- gate summary
- unsupported claim count
- required edits
- risk flags with severity and recommended fix

Human approval checklist:
- verify recipient and email
- verify source-backed claims
- edit or remove unsupported personalization
- approve only if gate verdict allows it
- do not claim reply/meeting outcomes without campaign data

Limitations:
- No reply-rate or meeting-rate claims yet
- Evidence confidence is source-presence based, not external verification
- Human review required before sending

3. Report tab in Streamlit.

In app/ui/layout.py:
- Add a new first tab named "Report".
- Current tabs are ["Sequence", "Research", "Contacts", "Drafts"].
- Change to ["Report", "Sequence", "Research", "Contacts", "Drafts"].
- In the Report tab, render the founder-readable report view and a Markdown download button.

Do not remove the existing top panels. Keep account score and quality gate near the top as they are now.

4. Report UI component.

In app/ui/components.py:
- Add a compact report component, e.g. report_panel(...), or several small helpers.
- The visual report should be cleaner than dumping raw Markdown.
- Suggested visible layout:
  - report hero card: company, priority, gate verdict, next action
  - decision grid: score, priority, gate, safe to send
  - two-column section: "Why this account" and "Human approval checklist"
  - evidence highlights
  - recommended contact
  - outreach sequence preview
  - risks and required edits
- Use existing components where reasonable, but avoid deeply nested cards.
- Keep it founder-friendly and readable in screenshots.

5. Download button.

In the Report tab:
- Use st.download_button to download the Markdown from build_account_report_markdown(...).
- File name should be safe and deterministic, e.g. bdr-account-report-globex-industries.md.
- The download should not write to disk.

6. Top evidence selection.

For both the Markdown and visual report, select top evidence deterministically:
- prefer safe_to_use=True
- prefer support_type="observed"
- prefer confidence_label high, then medium, then low
- prefer live_signal and job_signal before contact, manual_trigger, and icp_score
- include derived/internal ICP scoring only if useful, and label it clearly as internal/derived.

7. Recommended contact selection.

Pick the best contact deterministically:
- prefer contacts with email
- higher Hunter confidence
- title/persona relevance if available
- senior/executive seniority if useful

If no contact exists:
- report "No verified contact found"
- next action should mention manual contact discovery.

8. Next action logic.

Compute a deterministic next action for the report:
- If gate verdict is do_not_send_yet: "Do not send yet. Resolve risk flags and gather stronger evidence."
- Else if gate verdict is needs_more_research: "Gather stronger evidence before outreach."
- Else if gate verdict is needs_edit: "Edit required items, then re-review before sending."
- Else if gate verdict is approved but account priority is high_priority/review: "Human review, then approve/send manually."
- If no critic_result or no quality_gate: "Run critic or manually review before sending."

9. Styling.

In app/ui/theme.py:
- Add minimal Chalk-style CSS for report cards/sections.
- Use existing color tokens and 8px or smaller radius.
- Keep the page clean and screenshot-friendly.
- Make sure text does not overflow on desktop or mobile.

10. Safety and honesty.

- The report must clearly say manual review is required before sending.
- Do not imply the report is a guarantee of deliverability, replies, meetings, or revenue.
- Do not hide risk flags just because the sequence looks polished.
- Do not expose API keys, env vars, or private data.
- Source URLs are okay because they come from public evidence cards.
```
